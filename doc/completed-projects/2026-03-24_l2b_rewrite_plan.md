# qqrl ÔÇö Quick Query for RustLedger

Rewrite of [ledger2bql](https://github.com/alensiljak/ledger2bql) as a fast, single-binary Rust CLI that delegates all ledger parsing and BQL execution to an externally installed `rledger` binary.

---

## Goals

- Translate Ledger CLI query syntax into BQL, execute via `rledger query`, format output
- No Python runtime required
- Fast startup (~15ms vs ~1-2s for current Python stack)
- Single static binary, cross-compiles to Android ARM64 for Termux
- Depends on `rledger` being available on `$PATH` ÔÇö the user manages the rledger version independently
- License: AGPL-3.0

---

## Architecture

```txt
User input (CLI args)
        Ôöé
        Ôľ╝
qqrl (Rust CLI ÔÇö clap)
  ÔöťÔöÇ parse CLI args
  ÔöťÔöÇ build BQL query string  (same logic as current Python)
  ÔöťÔöÇ spawn: rledger query -f json $LEDGER_FILE "QUERY"
  ÔöťÔöÇ parse JSON response
  ÔöťÔöÇ format_output()  (running totals, hierarchy, lot arithmetic, etc.)
  ÔööÔöÇ tabulate Ôćĺ stdout  (pager skipped)
```

`rledger` handles: ledger file parsing, BQL execution, all financial arithmetic.
`qqrl` handles: CLI UX, BQL query construction, output formatting/tabulation.

## Design Decisions

### External binary, not linked library

- User upgrades `rledger` independently via their package manager (brew, scoop, cargo, pkg)
- qqrl has no compile-time dependency on rledger internals ÔÇö no Cargo.lock coupling
- Maintenance burden stays low after initial rewrite; improvements in speed/correctness come from rledger  
- Acceptable tradeoff: ~10-30ms subprocess overhead on Linux/Termux

### JSON output format

- `rledger query -f json FILE "QUERY"` returns structured data
- Avoids brittle text parsing; JSON schema is more stable than text table layout
- Must validate exact JSON schema in Phase 0 (field names, how Amount/Position are serialized)

### BQL query strings unchanged

- All `SELECT ... WHERE ... GROUP BY ... ORDER BY ... LIMIT` strings generated by the current Python code are valid BQL and run unchanged on rledger
- This is the key insight: only the I/O layer changes, not the query logic

## Commands

All 6 commands from ledger2bql are carried over:

| Command  | Alias | Description                           |
|----------|-------|---------------------------------------|
| `bal`    | `b`   | Account balances                      |
| `reg`    | `r`   | Transaction register                  |
| `query`  | `q`   | Execute named queries from .bean file |
| `lots`   | `l`   | Investment lots / cost basis          |
| `assert` | `a`   | Balance assertions                    |
| `price`  | `p`   | Price history                         |

Common options (all commands): `--begin`, `--end`, `--date-range`, `--sort`, `--limit`,
`--amount`, `--currency`, `--exchange`, `--total`

### Configuration

- `LEDGER_FILE` env var (or `$BEANCOUNT_FILE` for backward compat) ÔÇö path to .bean file
- `.env` file support via `dotenvy` crate
- `RLEDGER_BIN` env var to override the rledger binary path (default: `rledger`)

## Phased Plan

### Phase 0: Compatibility Audit ÔÇö *3-5 days, no code written*

Validate rledger behavior before committing to any implementation.

1. Install rledger, run `rledger query -f json tests/sample-ledger.bean "..."` for each query pattern used in ledger2bql (all 6 commands)
2. Document exact JSON schema ÔÇö field names, how `Amount`, `Position`, `Inventory` are serialized
3. Test these specifically (high risk ÔÇö not in rledger BQL docs):
   - `convert(position, 'EUR')` ÔÇö used by `--exchange/-X`
   - `value(position)` ÔÇö used by `lots`
   - `cost_number` column ÔÇö used by `lots`
   - `#balances` system table ÔÇö used by `assert`
   - `#prices` system table ÔÇö used by `price`
4. Check Android ARM64 release binary on GitHub Releases, or test `cargo install rustledger` in Termux
5. For any missing feature: either find the rledger equivalent syntax, or file an issue upstream
6. **Gate**: do not start Phase 1 until all 6 commands have a confirmed working query pattern

#### Revised Plan Based on Phase 0 Results

**Summary**: 4 of 6 commands are unblocked for Phase 1 work. The `lots` command is now implemented in qqrl for the supported `rledger` query shapes. The `assert` and `price` commands still depend on rledger's `#balances` and `#prices` system tables, which are pending feature requests.

**Unblocked commands** (implement in Phase 1):

- `bal` (balance) ÔÇö uses standard BQL, `convert(position, 'EUR')` works Ôťů
- `reg` (register) ÔÇö uses standard BQL Ôťů
- `query` (named queries) ÔÇö regex scan of .bean file Ôťů
- `lots` (investment lots) ÔÇö core command implemented; active lots, `--all`, `--average`, filters, sorting, and limit work Ôťů

**Blocked commands** (Phase 1.5 or Phase 2, pending rledger):

- `assert` (balance assertions) ÔÇö blocked on `#balances` system table ÔĆ│
- `price` (price history) ÔÇö blocked on `#prices` system table ÔĆ│

**Partially blocked behavior inside implemented commands**:

- `lots` market value column ÔÇö still blocked by current `rledger` `value(...)` behavior ÔĆ│

**Phase 1 Strategy**:

1. Implement full CLI skeleton with all 6 commands defined
2. Implement the 4 unblocked commands (bal, reg, query, lots)
3. Add placeholder error messages for `assert` and `price` that point to the rledger feature request
4. Ship Phase 1 with 4/6 commands working, noting the reduced `lots` output until `rledger` value support is usable

When rledger releases `#balances` and `#prices` support, implement Phase 1.5 to complete the remaining 2 commands (no architectural changes needed).

When `rledger` exposes a stable `value(...)` query shape for lots, extend `qqrl lots` to restore the market-value column without changing the CLI surface.

**Revised timeline**: ~3ÔÇô4 weeks for Phase 1 (instead of 4ÔÇô6 weeks)

### Phase 1: Rust CLI ÔÇö *~3-4 weeks*

Full rewrite. No intermediate Python step ÔÇö go straight to Rust.

#### Project setup

- `cargo new qqrl`
- Dependencies: `clap` (CLI), `serde_json` (JSON parsing), `dotenvy` (.env), `comfy-table` (tabulation)
- `AGPL-3.0` license

#### Core infrastructure

(start here, everything else depends on it)

1. `src/config.rs` ÔÇö resolve `LEDGER_FILE`, `RLEDGER_BIN`, `.env` loading
2. `src/runner.rs` ÔÇö `run_bql_query(query: &str) -> Result<Vec<serde_json::Value>>` ÔÇö spawns `rledger query -f json`, captures stdout, parses JSON, maps errors
3. `src/date_parser.rs` ÔÇö port of `date_parser.py`: `parse_date()`, `parse_date_range()`
4. `src/utils.rs` ÔÇö port of common utils: `parse_account_pattern()`, `parse_amount_filter()`, `parse_account_params()`
5. `src/cli.rs` ÔÇö clap command definitions, shared options struct

#### Migrate Commands

(can be worked in parallel after infrastructure is done)

6. `src/commands/balance.rs` ÔÇö `parse_query()` + `format_output()` including `--hierarchy` mode
7. `src/commands/register.rs` ÔÇö `parse_query()` + `format_output()` with running totals
8. `src/commands/query.rs` ÔÇö regex scan of .bean file for `query "name" "..."` directives (replaces `beancount.loader.load_file()`)
9. `src/commands/lots.rs` ÔÇö `parse_query()` + `format_output()` ÔÇö most complex; decimal arithmetic, depends on Phase 0 findings
10. `src/commands/assert.rs` ÔÇö `parse_query()` + `format_output()`
11. `src/commands/price.rs` ÔÇö `parse_query()` + `format_output()`

## Known Risks

| Risk                                          | Impact                             | Mitigation                                                                                      |
|-----------------------------------------------|------------------------------------|-------------------------------------------------------------------------------------------------|
| `convert(position, 'EUR')` not in rledger     | `--exchange/-X` broken             | Phase 0 test; workaround with two queries + post-processing, or file rledger issue              |

## What Carries Over Unchanged

From ledger2bql, these things translate directly:

- All BQL query strings (identical SQL dialect)
- Date parsing logic (`date_parser.py` Ôćĺ `date_parser.rs`, pure logic)
- Account pattern handling (`^pattern`, `pattern$`, raw regex)
- Amount filter parsing (`>100EUR`, `=50USD`, etc.)
- `--hierarchy` aggregation algorithm
- Running total tracking per currency in `reg`
- Named query regex pattern for `.bean` files

---

## What Changes

- `beanquery.connect().execute()` Ôćĺ `std::process::Command` spawning `rledger`
- Python beancount object access (`.units.number`, `.currency`) Ôćĺ `serde_json::Value` field access
- `beancount.loader.load_file()` Ôćĺ regex scan of `.bean` file text
- `tabulate()` Ôćĺ `comfy-table` or equivalent Rust crate
- `click` Ôćĺ `clap`
- `loguru` Ôćĺ `tracing` (optional, for `--verbose`)
- `dotenv` Ôćĺ `dotenvy`
