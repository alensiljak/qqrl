# qqrl ‚Äî Quick Query for RustLedger

Rewrite of [ledger2bql](https://github.com/alensiljak/ledger2bql) as a fast, single-binary Rust CLI that delegates all ledger parsing and BQL execution to an externally installed `rledger` binary.

---

## Goals

- Translate Ledger CLI query syntax into BQL, execute via `rledger query`, format output
- No Python runtime required
- Fast startup (~15ms vs ~1-2s for current Python stack)
- Single static binary, cross-compiles to Android ARM64 for Termux
- Depends on `rledger` being available on `$PATH` ‚Äî the user manages the rledger version independently
- License: AGPL-3.0

---

## Architecture

```txt
User input (CLI args)
        ‚îÇ
        ‚ñº
qqrl (Rust CLI ‚Äî clap)
  ‚îú‚îÄ parse CLI args
  ‚îú‚îÄ build BQL query string  (same logic as current Python)
  ‚îú‚îÄ spawn: rledger query -f json $LEDGER_FILE "QUERY"
  ‚îú‚îÄ parse JSON response
  ‚îú‚îÄ format_output()  (running totals, hierarchy, lot arithmetic, etc.)
  ‚îî‚îÄ tabulate ‚Üí stdout  (pager skipped)
```

`rledger` handles: ledger file parsing, BQL execution, all financial arithmetic.
`qqrl` handles: CLI UX, BQL query construction, output formatting/tabulation.

## Design Decisions

### External binary, not linked library

- User upgrades `rledger` independently via their package manager (brew, scoop, cargo, pkg)
- qqrl has no compile-time dependency on rledger internals ‚Äî no Cargo.lock coupling
- Maintenance burden stays low after initial rewrite; improvements in speed/correctness come from rledger  
- Acceptable tradeoff: ~10-30ms subprocess overhead on Linux/Termux

### JSON output format

- `rledger query -f json FILE "QUERY"` returns structured data
- Avoids brittle text parsing; JSON schema is more stable than text table layout
- Must validate exact JSON schema in Phase 0 (field names, how Amount/Position are serialized)

### BQL query strings unchanged

- All `SELECT ... WHERE ... GROUP BY ... ORDER BY ... LIMIT` strings generated by the current Python code are valid BQL and run unchanged on rledger
- This is the key insight: only the I/O layer changes, not the query logic

## Commands

All 6 commands from ledger2bql are carried over:

| Command  | Alias | Description                           |
|----------|-------|---------------------------------------|
| `bal`    | `b`   | Account balances                      |
| `reg`    | `r`   | Transaction register                  |
| `query`  | `q`   | Execute named queries from .bean file |
| `lots`   | `l`   | Investment lots / cost basis          |
| `assert` | `a`   | Balance assertions                    |
| `price`  | `p`   | Price history                         |

Common options (all commands): `--begin`, `--end`, `--date-range`, `--sort`, `--limit`,
`--amount`, `--currency`, `--exchange`, `--total`

### Configuration

- `LEDGER_FILE` env var (or `$BEANCOUNT_FILE` for backward compat) ‚Äî path to .bean file
- `.env` file support via `dotenvy` crate
- `RLEDGER_BIN` env var to override the rledger binary path (default: `rledger`)

## Phased Plan

### Phase 0: Compatibility Audit ‚Äî *3-5 days, no code written*

Validate rledger behavior before committing to any implementation.

1. Install rledger, run `rledger query -f json tests/sample-ledger.bean "..."` for each query pattern used in ledger2bql (all 6 commands)
2. Document exact JSON schema ‚Äî field names, how `Amount`, `Position`, `Inventory` are serialized
3. Test these specifically (high risk ‚Äî not in rledger BQL docs):
   - `convert(position, 'EUR')` ‚Äî used by `--exchange/-X`
   - `value(position)` ‚Äî used by `lots`
   - `cost_number` column ‚Äî used by `lots`
   - `#balances` system table ‚Äî used by `assert`
   - `#prices` system table ‚Äî used by `price`
4. Check Android ARM64 release binary on GitHub Releases, or test `cargo install rustledger` in Termux
5. For any missing feature: either find the rledger equivalent syntax, or file an issue upstream
6. **Gate**: do not start Phase 1 until all 6 commands have a confirmed working query pattern

### Revised Plan Based on Phase 0 Results

**Summary**: 4 of 6 commands are unblocked for Phase 1 work. The `lots` command is now implemented in qqrl for the supported `rledger` query shapes. The `assert` and `price` commands still depend on rledger's `#balances` and `#prices` system tables, which are pending feature requests.

**Unblocked commands** (implement in Phase 1):

- `bal` (balance) ‚Äî uses standard BQL, `convert(position, 'EUR')` works ‚úÖ
- `reg` (register) ‚Äî uses standard BQL ‚úÖ
- `query` (named queries) ‚Äî regex scan of .bean file ‚úÖ
- `lots` (investment lots) ‚Äî core command implemented; active lots, `--all`, `--average`, filters, sorting, and limit work ‚úÖ

**Blocked commands** (Phase 1.5 or Phase 2, pending rledger):

- `assert` (balance assertions) ‚Äî blocked on `#balances` system table ‚è≥
- `price` (price history) ‚Äî blocked on `#prices` system table ‚è≥

**Partially blocked behavior inside implemented commands**:

- `lots` market value column ‚Äî still blocked by current `rledger` `value(...)` behavior ‚è≥

**Phase 1 Strategy**:

1. Implement full CLI skeleton with all 6 commands defined
2. Implement the 4 unblocked commands (bal, reg, query, lots)
3. Add placeholder error messages for `assert` and `price` that point to the rledger feature request
4. Ship Phase 1 with 4/6 commands working, noting the reduced `lots` output until `rledger` value support is usable

When rledger releases `#balances` and `#prices` support, implement Phase 1.5 to complete the remaining 2 commands (no architectural changes needed).

When `rledger` exposes a stable `value(...)` query shape for lots, extend `qqrl lots` to restore the market-value column without changing the CLI surface.

**Revised timeline**: ~3‚Äì4 weeks for Phase 1 (instead of 4‚Äì6 weeks)

### Phase 1: Rust CLI ‚Äî *~3-4 weeks*

Full rewrite. No intermediate Python step ‚Äî go straight to Rust.

#### Project setup

- `cargo new qqrl`
- Dependencies: `clap` (CLI), `serde_json` (JSON parsing), `dotenvy` (.env), `comfy-table` (tabulation)
- `AGPL-3.0` license

#### Core infrastructure

(start here, everything else depends on it)

1. `src/config.rs` ‚Äî resolve `LEDGER_FILE`, `RLEDGER_BIN`, `.env` loading
2. `src/runner.rs` ‚Äî `run_bql_query(query: &str) -> Result<Vec<serde_json::Value>>` ‚Äî spawns `rledger query -f json`, captures stdout, parses JSON, maps errors
3. `src/date_parser.rs` ‚Äî port of `date_parser.py`: `parse_date()`, `parse_date_range()`
4. `src/utils.rs` ‚Äî port of common utils: `parse_account_pattern()`, `parse_amount_filter()`, `parse_account_params()`
5. `src/cli.rs` ‚Äî clap command definitions, shared options struct

#### Migrate Commands

(can be worked in parallel after infrastructure is done)

6. `src/commands/balance.rs` ‚Äî `parse_query()` + `format_output()` including `--hierarchy` mode
7. `src/commands/register.rs` ‚Äî `parse_query()` + `format_output()` with running totals
8. `src/commands/query.rs` ‚Äî regex scan of .bean file for `query "name" "..."` directives (replaces `beancount.loader.load_file()`)
9. `src/commands/lots.rs` ‚Äî `parse_query()` + `format_output()` ‚Äî most complex; decimal arithmetic, depends on Phase 0 findings
10. `src/commands/assert.rs` ‚Äî `parse_query()` + `format_output()`
11. `src/commands/price.rs` ‚Äî `parse_query()` + `format_output()`

### Phase 1 status snapshot

- `balance` implemented and covered by integration tests
- `register` implemented and covered by integration tests
- `query` implemented
- `lots` implemented with a reduced output shape that omits market value for now
- `assert` placeholder implemented
- `price` placeholder implemented

### Lots status

Implemented in qqrl today:

- dedicated `lots` CLI options: `--all`, `--average`, `--sort-by`
- default active-lot view using grouped positions with positive remaining quantity
- detailed lot view with `--all`
- average-cost view with `--average`
- account, date, currency, and amount filters
- sort and limit support
- integration coverage in `tests/lots_tests.rs`

Still pending for full parity:

- market value / `Value` column in output
- any exchange-driven valuation for lots

Current blocker details:

- `value(position)` currently fails for the detailed query shape unless a target currency is provided
- `value(SUM(position))` returns an inventory-like structure in current testing rather than a directly displayable market-value amount
- until that behavior is fixed or clarified upstream, `lots` uses `price` and `cost` columns only

## What Carries Over Unchanged

From ledger2bql, these things translate directly:

- All BQL query strings (identical SQL dialect)
- Date parsing logic (`date_parser.py` ‚Üí `date_parser.rs`, pure logic)
- Account pattern handling (`^pattern`, `pattern$`, raw regex)
- Amount filter parsing (`>100EUR`, `=50USD`, etc.)
- `--hierarchy` aggregation algorithm
- Running total tracking per currency in `reg`
- Named query regex pattern for `.bean` files

---

## What Changes

- `beanquery.connect().execute()` ‚Üí `std::process::Command` spawning `rledger`
- Python beancount object access (`.units.number`, `.currency`) ‚Üí `serde_json::Value` field access
- `beancount.loader.load_file()` ‚Üí regex scan of `.bean` file text
- `tabulate()` ‚Üí `comfy-table` or equivalent Rust crate
- `click` ‚Üí `clap`
- `loguru` ‚Üí `tracing` (optional, for `--verbose`)
- `dotenv` ‚Üí `dotenvy`
